# コンテンツ生成ワークフロー（汎用版）

プロジェクトのWebサイト向けに、SEO/LLMOを意識したコンテンツを生成する。
初回はヒアリングで設定ファイルを作成し、以降はその設定に基づいて5段階パイプラインを実行する。

---

## Phase 0: 設定ファイルの確認

### 最初に `docs/content-config.md` を探す

- **存在する** → 読み込んで Phase 2 へ進む
- **存在しない** → Phase 1（ヒアリング）を実行する

### 補足: `docs/seo-config.md` が存在する場合

`/seo-check` で作成済みの `docs/seo-config.md` があれば、ターゲット定義・キーワード等を引き継ぐ。
ヒアリング時に該当項目はスキップし、「seo-config.md から引き継ぎました」と伝える。

---

## Phase 1: 初回ヒアリングシーケンス

設定ファイルがない場合、以下の質問を**1問ずつ順番に**ユーザーに聞く。
全回答を得たら `docs/content-config.md` を生成して保存する。

### 質問リスト

```
Q1: プロジェクト名とサイトURLを教えてください。
    （seo-config.md があればスキップ）

Q2: ターゲットを教えてください。
    - 年齢層、属性、地域、心理状態・悩み
    （seo-config.md があればスキップ）

Q3: コンテンツの形式は何ですか？
    （例: ブログ / マガジン / コラム / ガイド / お知らせ / 事例紹介）

Q4: コンテンツのカテゴリ体系を教えてください。
    決まっていなければ「未定」でOK — 業種とターゲットから提案します。
    （例: 「施工事例」「お役立ち情報」「スタッフブログ」）

Q5: ブランドボイス（文章のトーン）はどうしますか？
    a) フォーマル（です/ます体・専門的）
    b) カジュアル（親しみやすい・語りかけ調）
    c) 専門的だけど読みやすい（データ重視・平易な表現）
    d) その他（具体的に）

    また、文章で絶対にやってはいけないこと（NGルール）はありますか？
    （例: 「！の多用禁止」「英語カタカナ語を避ける」「業界用語をそのまま使わない」）

Q6: 記事の末尾にどんなCTA（行動喚起）を置きますか？
    （例: お問い合わせフォーム / 資料請求 / 電話番号 / エントリーページ / 無料相談）
    リンク先のURLも教えてください。

Q7: 画像はどうしますか？
    a) AI生成する（Nanobanana使用）→ スタイルの方向性も教えてください
    b) 素材写真を使う（自前/ストックフォト）
    c) 画像なしでOK
    d) その他

    AI生成の場合、アンカー画像（スタイルの基準になる画像）はありますか？
    パスがあれば教えてください。

Q8: コンテンツのファイル管理方式を教えてください。
    a) Markdown（ディレクトリパスを指定）
    b) CMS（種類を指定）
    c) まだ決まっていない → 提案します

    Markdownの場合:
    - 記事の保存先（例: src/content/blog/）
    - データ定義ファイル（例: src/data/blog-data.ts）
    - 画像の保存先（例: public/img/blog/）
```

### 設定ファイル生成

全回答を得たら `docs/content-config.md` を作成する:

```markdown
---
project_name: "[プロジェクト名]"
site_url: "[URL]"
content_type: "[コンテンツ形式]"
created: "YYYY-MM-DD"
---

# コンテンツ生成設定

## ターゲット定義

- **年齢層**: [回答]
- **属性**: [回答]
- **地域**: [回答]
- **心理状態・悩み**: [回答]

## カテゴリ体系

| カテゴリ | テーマ | 色（任意） |
|---------|--------|-----------|
| [カテゴリ1] | [テーマ] | [色] |
| [カテゴリ2] | [テーマ] | [色] |

（未定の場合はヒアリング内容から提案して記載。ユーザー承認を得ること。）

## ブランドボイス

- **トーン**: [回答]
- **口調ルール**:
  - 見出し: [語りかけ調 / 体言止め / 疑問形 等]
  - 本文: [です/ます体 / だ/である体]
  - 温度感: [ターゲットが読んでどう感じるべきか]
- **NGルール**:
  - [回答]

## CTA設計

- **メインCTA**: [テキスト] → [URL]
- **サブCTA**: [テキスト] → [URL]（あれば）

## 画像生成

- **方式**: [AI生成 / 素材写真 / なし]
- **スタイル定数**（AI生成の場合）:
```python
STYLE = "[ヒアリング内容から生成したスタイルプロンプト]"
```
- **アンカー画像**: [パス or なし]
- **アスペクト比**: サムネイル [3:2] / ヒーロー [16:9]
- **出力先**: [パス]

## ファイル管理

- **記事保存先**: [パス]
- **データ定義**: [パス]
- **画像保存先**: [パス]
- **テンプレートファイル**: [パス]（あれば）

## 実装時に更新するファイル一覧

| ファイル | 更新内容 |
|---------|---------|
| [記事保存先]/[slug].md | 記事本体 |
| [データ定義] | カード情報追加 |
| [画像保存先]/[slug].png | サムネイル画像 |
| sitemap.ts | URLエントリー追加（動的なら不要） |
| public/ai.txt | コンテンツ情報追記 |
| public/llms.txt | コンテンツ情報追記 |

## コンテンツ生成履歴

| slug | タイトル | カテゴリ | 日付 |
|------|---------|---------|------|
```

設定ファイルを保存したら、「設定を保存しました。コンテンツを作成しますか？」と聞く。

---

## Phase 2: コンテンツ生成パイプライン

`docs/content-config.md` を読み込み、5段階パイプラインを実行する。

### Step 0: テーマ決定

ユーザーがテーマを指定している場合 → そのまま Step 1 へ。
指定がない場合 → 以下から提案する:

1. `docs/seo-config.md` のキーワードギャップ（あれば）
2. ターゲットの悩みから逆算したテーマ候補
3. カテゴリ間のバランス（生成履歴を確認して偏りを防ぐ）

**テーマ提案フォーマット:**
```
| # | テーマ案 | カテゴリ | 狙うキーワード | 理由 |
|---|---------|---------|--------------|------|
| 1 | ... | ... | ... | ... |
| 2 | ... | ... | ... | ... |
| 3 | ... | ... | ... | ... |
```

ユーザーの承認を得てから次へ。

---

### Step 1: リサーチ

```
Agent: general-purpose（WebSearch/WebFetch使用）
※ フォアグラウンドで実行すること

目的: テーマについて正確な事実を集め、裏どりし、コンテンツの土台を作る。

手順:
1. テーマに関連する統計・研究・事例をWebSearchで収集
2. 各データにWebFetchでアクセスし、裏を取る
3. ターゲットの視点で再構成する（「この数字がターゲットにとって何を意味するか」）

出典の記録（1件につき5項目すべて必須）:
- 組織名
- 資料名
- URL
- 年度/更新日
- 具体的な数値とコンテキスト

絶対禁止:
- 出典のない数値・事実
- 推測表現（「一般的に」「よく言われる」）
- AIの知識を出典なしで記載

出力: リサーチ結果（裏どり済み事実一覧 + 出典 + ターゲットにとっての意味）
```

---

### Step 2: 構成設計

```
目的: リサーチ結果を受け取り、誰に・何を・どの順で伝えるかを設計する。

手順:
1. 「読んだ後にターゲットが『できる』こと」を1文で定義
2. content-config.md のブランドボイスに合った構成パターンを選択
3. セクション構成を設計（各セクションに「目的」「内容概要」「感情段階」を明記）
4. 感情の流れを確認（共感→気づき→希望→行動）
5. CTA配置ポイントを設計

出力フォーマット:
| # | セクション名 | 目的 | 内容概要 | 感情段階 |
|---|------------|------|---------|---------|

※ 構成案はユーザーに提示し、承認を得てから次へ進む
```

---

### Step 3: 執筆

```
目的: 構成案とリサーチ結果を受け取り、ターゲットが読んで動けるコンテンツを書く。

ルール（content-config.md から読み込む）:
- ブランドボイス（トーン・口調・NGルール）を厳守
- 全ての数値に出典（調査名・発表元・年度）を付ける
- 推測表現禁止
- テンプレ文言禁止（「成長できる環境」「風通しの良い」等）
- CTA配置は構成設計に従う

出力: frontmatter付きMarkdownファイル

frontmatterテンプレート:
---
title: "記事タイトル"
date: "YYYY-MM-DD"
category: "[カテゴリ]"
excerpt: "120文字以内の概要（meta descriptionにも使用）"
tags: ["タグ1", "タグ2"]
readingTime: "○分"
image: "[画像パス]"
---

品質チェック（セルフレビュー）:
- ターゲットが読んだ後に具体的な行動を取れるか？
- ブランドボイスに沿っているか？
- 出典は全て明記されているか？
- CTAが自然に配置されているか？
```

---

### Step 4: 実装

```
目的: 執筆済みコンテンツを受け取り、ファイル配置・画像生成・データ登録を行う。

手順（content-config.md の「実装時に更新するファイル一覧」に従う）:

1. Markdownファイルを指定パスに配置
2. データ定義ファイルにカード情報を追加
3. 画像を生成（AI生成の場合）
   - content-config.md のスタイル定数を使用
   - アンカー画像がある場合は1枚だけ参照（複数アンカー禁止）
   - 画像生成にはNanobananaを使用（docs/NANOBANANA.md参照）
4. sitemap更新（動的でない場合）
5. ai.txt にコンテンツ情報を追記
6. llms.txt にコンテンツ情報を追記
7. インデックス登録手順書にURL追記（ファイルが存在する場合のみ）

チェックリスト:
- [ ] 記事Markdownファイル配置済み
- [ ] データ定義ファイル更新済み
- [ ] サムネイル画像配置済み
- [ ] ai.txt 追記済み
- [ ] llms.txt 追記済み
```

---

### Step 5: QA

```
目的: 完成したコンテンツを複数の観点から検証する。

チェック1: 品質基準
- ターゲットが読んだ後に「次に何をすべきか」わかるか？
- ブランドボイスが自然に体現されているか？
- 「どこにでもある一般論」になっていないか？
- CTAへの感情の流れがあるか？

チェック2: コンテンツ品質
- 全ての数値に出典があるか
- 推測表現がないか
- テンプレ文言がないか
- 見出しがcontent-config.mdの口調ルールに合っているか

チェック3: Markdown品質
- frontmatterが完全か
- excerptが120文字以内か
- 見出し階層が適切か（h2→h3、h1不使用）

チェック4: データ登録
- content-config.md の「実装時に更新するファイル一覧」が全て更新されているか

チェック5: SEO
- excerptがmeta descriptionとして適切か
- tagsにSEOキーワードが含まれているか

1つでも不合格なら修正を要求する。
全合格したら content-config.md の「コンテンツ生成履歴」に追記する。
```

---

## 注意事項

- **設定ファイル依存**: 全ての判断は `docs/content-config.md` に基づく。設定にないことは勝手に決めず、ユーザーに聞く
- **承認ポイント**: Step 0（テーマ選定）と Step 2（構成設計）は必ずユーザー承認を得てから次へ進む
- **WebSearch必須**: Step 1（リサーチ）は `subagent_type: "general-purpose"` を使う
- **フォアグラウンド実行**: WebSearch系はバックグラウンドで実行しない
- **出典厳守**: 出典のない数値・事実は一切使わない
- **seo-config.md との連携**: 存在する場合、ターゲット・キーワードを共有する。コンテンツ追加提案があればそれをテーマ候補にする

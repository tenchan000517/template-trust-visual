# コンテンツ生成ワークフロー

Webサイト向けSEO/LLMOコンテンツを生成する5段階パイプライン。
初回はヒアリングで設定ファイルを作成し、以降は設定に従ってリサーチ→構成設計→執筆→実装→QAを実行する。

---

## Phase 0: 設定ファイルの確認

`docs/content-config.md` を探す。

- **存在する** → 読み込んで Phase 2 へ
- **存在しない** → Phase 1（ヒアリング）へ

`docs/seo-config.md` が存在する場合、ターゲット・キーワード等はそこから引き継ぐ。

---

## Phase 1: 初回ヒアリング

設定ファイルがない場合、以下を**1問ずつ順番に**聞く。全回答を得たら `docs/content-config.md` を生成する。

### 質問リスト

```
Q1: プロジェクト名とサイトURLを教えてください。
    （seo-config.md があればスキップ）

Q2: ターゲットを教えてください。
    - 年齢層、属性、地域
    - 心理状態・悩み（例: 「リフォームしたいけど業者選びがわからない」）
    （seo-config.md があればスキップ）

Q3: コンテンツの名前を教えてください。
    （例: ブログ / コラム / ガイド / お役立ち情報 / スタッフブログ）

Q4: ハブページ（記事一覧ページ）のURLはどこにしますか？
    （例: /blog, /column, /guide, /useful-info）

Q5: カテゴリ体系を教えてください。
    決まっていなければ「未定」でOK — 業種とターゲットから提案します。
    （例: 「施工事例」「お役立ち情報」「業界ニュース」）

Q6: ブランドボイス（文章のトーン）はどうしますか？
    a) フォーマル（です/ます体・専門的）
    b) カジュアル（親しみやすい・語りかけ調）
    c) 専門的だけど読みやすい（データ重視・平易な表現）
    d) その他（具体的に）

    文章で絶対にやってはいけないこと（NGルール）はありますか？
    （例: 「！の多用禁止」「業界用語をそのまま使わない」）

Q7: 記事の末尾にどんなCTAを置きますか？
    （例: お問い合わせ / 資料請求 / エントリー / 無料相談）
    リンク先のURLも教えてください。

Q8: アンカー画像（画像スタイルの基準になる画像）はありますか？
    あればパスを教えてください。なければAI生成時にスタイルを一から作ります。
    ※ 画像は基本的にAI生成します（Nanobanana使用）
```

## Phase 2: コンテンツ生成パイプライン

`docs/content-config.md` を読み込み、5段階で実行する。

### Step 0: テーマ決定

ユーザーがテーマを指定している場合 → Step 1 へ。
指定がない場合 → 以下から提案:

1. `docs/seo-config.md` のコンテンツ追加提案（あれば）
2. ターゲットの悩みから逆算したテーマ候補
3. カテゴリ間のバランス（生成履歴を確認して偏り防止）

テーマ提案はテーブルで提示し、**ユーザー承認を得てから**次へ。

---

### Step 1: リサーチ

> **エージェントタイプ**: `subagent_type: "general-purpose"`（WebSearch/WebFetch使用）
> **バックグラウンド実行禁止**: 権限プロンプトが表示されず自動拒否されるため

#### リサーチの姿勢

リサーチした内容をそのまま使うのではない。
リサーチした内容から**ターゲットにとって本当に大事なこと**を抽出し、ターゲットの気持ちになって伝える形に翻訳する。

#### 4ステップ

1. **事実を集める** — 統計、研究結果、企業事例、専門家の見解
2. **裏を取る** — 公式ソースで確認。間違いは許されない
3. **ターゲットの視点で再構成する** — 「ターゲットの自分が読んで響くか？動けるか？」で判断
4. **自社の強みとの接点を見つける** — 事実と自社の哲学・サービスが接続できる切り口を探す

#### 絶対禁止

- **出典のない数値・事実** — 1つたりとも許さない
- **元ネタのURLを裏どりせずに転記** — 必ずWebFetch/WebSearchして確認する
- **「一般的に」「よく言われる」「〜でしょう」等の推測表現** — 事実か、出典があるか、でなければ書くな
- **AIの知識を出典なしで記載** — 自分の記憶は出典ではない

#### 裏どりの手順

全てのデータについて、以下を1件ずつ実行:

1. **アクセス**: 出典URLにWebFetchでアクセス。404でないか確認
2. **データ確認**: 該当する数値・事実が実際に記載されているか確認。二次情報の場合は可能なら一次ソースまで遡る
3. **メタデータ記録**: 年度/更新日を確認（原則3年以内）。調査対象・母集団を確認。複数ソースで矛盾がないか確認
4. **判定**: 裏が取れた→使用OK。取れない→使用しない

#### 出典の記録フォーマット（1件につき5項目すべて必須）

```
- 組織名: （例: 総務省）
- 資料名: （例: 労働力調査）
- URL: （例: https://www.stat.go.jp/...）
- 年度/更新日: （例: 2025年公表）
- 具体的な数値とコンテキスト: （例: 20代の転職率 ○○%、業界別内訳あり）
```

#### リサーチ結果の出力フォーマット

```markdown
# リサーチ結果: [テーマ名]

## 1. テーマの概要

## 2. 裏どり済み事実一覧
（各項目に: 事実 / 出典5項目 / ターゲットにとっての意味 / 自社との接続案）

## 3. 出典一覧（簡易版）

## 4. 裏どり不十分な項目（理由を明記）

## 5. ターゲットの疑問への回答案

## 6. 自社の強みとの接続ポイント

## 7. 追加リサーチが必要な項目
```

---

### Step 2: 構成設計

#### 手順

1. **「読んだ後にターゲットが『できる』こと」を1文で定義** — これが構成全体の軸。曖昧なら全体がぶれる
2. **content-config.md のブランドボイスに合った構成パターンを選択**
3. **セクション構成を設計** — 各セクションに「目的」「内容概要」「感情段階」を明記
4. **感情の流れを確認** — 共感→気づき→希望→行動
5. **CTA配置を設計** — 感情が高まったポイントに置く。末尾に機械的に置くだけでは効果薄い

#### 構成パターン

**パターン1: マインドセット型**（意識・考え方系）
```
1. フック（読者の悩み・問いかけ）
2. 視点の転換（新しい見方を提示）
3. 自社の事例 or 具体的な行動指針
4. まとめ（次の一歩を明確に）
5. CTA
```

**パターン2: ハウツー型**（やり方・手順系）
```
1. 課題の提示（ターゲットが直面している壁）
2. 解決のフレームワーク（3-5ステップ）
3. 各ステップの詳細（データ or 事例付き）
4. よくある失敗パターン
5. まとめ + CTA
```

**パターン3: 事例型**（実績・ストーリー系）
```
1. 課題・状況の提示
2. 自社の具体的な事例・エピソード
3. そこから導かれる原則
4. 読者の状況への応用
5. CTA
```

#### 出力フォーマット

```markdown
## 構成案: [タイトル]

**カテゴリ**: [カテゴリ]
**URL**: [hub_url]/[slug]
**使用パターン**: マインドセット型 / ハウツー型 / 事例型

### 読んだ後に「できる」こと
[1文で]

### SEOキーワード
[メイン1 + サブ2-3]

### セクション構成

| # | セクション名 | 目的 | 内容概要 | 感情段階 |
|---|------------|------|---------|---------|

### CTA設計
[どの感情ポイントでどのCTAを配置するか]
```

**構成案はユーザーに提示し、承認を得てから次へ進む。**

---

### Step 3: 執筆

#### ルール（content-config.md から読み込む）

- **ブランドボイス（トーン・口調・NGルール）を厳守**
- 全ての数値に出典（調査名・発表元・年度）を付ける
- 推測表現禁止（「一般的に」「よく言われる」）
- テンプレ文言禁止（「成長できる環境」「風通しの良い」「お客様に寄り添い」等）
- h1は使わない（h2から開始）
- Markdown構造を活かす（##で大テーマ、###でサブ、リストで要点、>で印象的フレーズ、**太字**で核心）

#### 数字の扱い

- 数字は文脈に埋め込む（「○○%」で終わらない）
- 年度・調査対象は必ず明記する
- 過去の日付を未来形で書かない

#### 出力形式

frontmatter付きMarkdownファイル:

```yaml
---
title: "記事タイトル"
date: "YYYY-MM-DD"
category: "[カテゴリ]"
excerpt: "120文字以内の概要（meta descriptionにも使用）"
featured: false
tags: ["タグ1", "タグ2"]
readingTime: "○分"
image: "/img/[content_name]/[slug].png"
---
```

#### セルフレビュー

書き終えたら以下を確認:
- [ ] ターゲットが読んだ後に具体的な行動を取れるか？
- [ ] ブランドボイスに沿っているか？
- [ ] 出典は全て明記されているか？推測表現はないか？
- [ ] content-config.md の品質基準5項目に全てYesと言えるか？

---

### Step 4: 実装

#### 手順（content-config.md の「実装時に更新するファイル一覧」に従う）

1. **Markdownファイルを配置** — `src/content/[name]/[slug].md`
2. **データ定義ファイルにカード情報を追加** — `available: true`
3. **画像を生成**
   - content-config.md のスタイル定数を使用
   - アンカー画像がある場合は**1枚だけ**参照（複数アンカー禁止）
   - Nanobananaで生成（`docs/NANOBANANA.md` 参照）
   - 生成後にReadツールで目視確認
4. **sitemap更新**（動的でない場合）
5. **ai.txt にコンテンツ情報を追記**
6. **llms.txt にコンテンツ情報を追記**
7. **インデックス登録手順書にURL追記**（ファイルが存在する場合のみ）

#### 画像生成スクリプトの書き方

```python
import os, sys, time
sys.path.insert(0, os.path.dirname(os.path.abspath(__file__)))
from generate_image_pro import generate_with_reference_pro

STYLE = "[content-config.mdのスタイル定数]"
ANCHOR = r"[content-config.mdのアンカー画像パス]"  # 1枚だけ
OUTPUT = r"[content-config.mdの画像出力先]"

generate_with_reference_pro(
    f"{STYLE} [記事テーマに合った描写]",
    [ANCHOR],
    os.path.join(OUTPUT, "[slug].png"),
    "3:2"
)
```

実行: `cmd.exe /c "cd C:\Nanobanana && C:\Nanobanana\venv\Scripts\python.exe tmp_script.py"`
実行後にtmpスクリプトは削除する。

### Step 5: QA

#### チェック1: 品質基準

content-config.md の品質基準に全てYesと言えなければ不合格。

#### チェック2: コンテンツ品質

- [ ] 読んだターゲットが「次に何をすればいいか」わかるか
- [ ] 全ての数値に出典（調査名・発表元・年度）があるか
- [ ] 推測表現が含まれていないか
- [ ] ブランドボイス（口調・トーン・NGルール）に沿っているか
- [ ] テンプレ文言が使われていないか

#### チェック3: Markdown品質

- [ ] frontmatterが完全か（title, date, category, excerpt, tags, readingTime, image）
- [ ] excerptが120文字以内か
- [ ] 見出し階層が適切か（h2→h3、h1不使用）

#### チェック4: データ登録

- [ ] content-config.md の「実装時に更新するファイル一覧」が全て更新されているか

#### チェック5: SEO

- [ ] excerptがmeta descriptionとして適切か
- [ ] tagsにSEOキーワードが含まれているか

**1つでも不合格なら修正を要求する。**
全合格したら content-config.md の「コンテンツ生成履歴」に追記する。

---

## 注意事項

- **設定ファイル依存**: 全ての判断は `docs/content-config.md` に基づく。設定にないことは勝手に決めず、ユーザーに聞く
- **承認ポイント**: Step 0（テーマ選定）と Step 2（構成設計）は必ずユーザー承認を得てから進む
- **WebSearch必須**: Step 1（リサーチ）は `subagent_type: "general-purpose"` を使う。Explore/Planタイプは不可
- **フォアグラウンド実行**: WebSearch系はバックグラウンドで実行しない
- **出典厳守**: 出典のない数値・事実は一切使わない。AIの記憶は出典ではない
- **seo-config.md との連携**: 存在する場合、ターゲット・キーワードを共有しコンテンツ追加提案をテーマ候補にする
